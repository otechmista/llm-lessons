The Training Loop: Where Learning Happens

All the pieces are in place. The training loop ties them together and runs the whole process from start to finish.

Where This Lives

app/train.py

What Happens, Step by Step

Run python app/train.py and this is what executes:

  1. Load the dataset: read app/dataset.txt into memory
  2. Build the vocabulary: collect every unique character from the text
  3. Encode the text: convert all characters to token IDs
  4. Create the model: build a fresh MiniGPT with random weights
  5. Create the optimizer: set up AdamW to handle weight updates

Then the loop begins:

  1. Sample a random batch: pick random windows of text from the encoded dataset
  2. Forward pass: feed the batch through the model to get logits
  3. Compute loss: compare predictions against the correct next tokens
  4. Backpropagate: trace the error back through the network, compute gradients
  5. Update weights: AdamW adjusts every parameter slightly
  6. Log progress: print the current loss
  7. Repeat: go back to step 6

After all steps complete:

  1. Save a checkpoint: store the model weights, vocabulary, and config to disk

What Is a Batch?

Instead of training on one example at a time, we train on many simultaneously. A batch is a group of training examples processed together. This is faster and produces more stable gradients.

In app/config.py:

batch_size = 16   (process 16 examples at once)
block_size = 64   (each example is 64 characters long)

A batch is like a small pack of flashcards. The model studies 16 tiny windows at once, averages the learning signal, and makes one update. This is faster than one example at a time and usually less noisy.

Epochs, Steps, and Gradient Updates

A step is one optimizer update. An epoch means one full pass over the training data. In this project, we sample random windows for a fixed number of steps, which is easier for a tiny study model than tracking exact epochs.

TermBeginner meaningIn code
BatchA pack of examples studied togetherx, y = get_batch(...)
StepOne weight updateoptimizer.step()
EpochOne pass through the datasetConceptual here, not explicitly counted
x, y = get_batch(data)
logits = model(x)
loss = compute_loss(logits, y)
loss.backward()
optimizer.step()
optimizer.zero_grad()

Common mistake: bigger batch is not automatically better. Bigger batches use more memory and can sometimes learn less flexibly. Production systems tune batch size carefully.

Pretraining, Fine-Tuning, and RLHF

This project is closest to a tiny pretraining run: it reads a text corpus and learns next-token prediction. Commercial assistants usually add more stages after that.

StageWhat it teachesAnalogy
PretrainingGeneral language patternsSchool
Fine-tuningA domain, style, or taskJob training
Instruction tuningHow to follow user requestsPractice conversations
RLHFPreference-shaped behaviorCoaching for helpfulness
pretrain(general_corpus)
finetune(instruction_examples)
evaluate_with_human_preferences()

Important distinction: fine-tuning changes model weights. RAG, which appears later, gives the model retrieved notes at answer time without changing weights.

Try changing max_steps to train longer and watch if loss goes lower.

Why the Dataset Is Small

Training on a small dataset is intentional. The goal is study, not power. Small data = fast training = easy to experiment and observe. You can retrain from scratch in under a minute.
