The API Layer: Talking to the Model Over HTTP

The model can answer questions from the command line. But what if you want to build a web app, a chatbot interface, or connect it to other tools?

That's where the API comes in. It wraps the model in an HTTP server that anyone can send requests to — just like the real OpenAI API.

Where This Lives

app/api.py
app/schemas.py

Start the Server

First, make sure you've trained the model:

python app/train.py

Then start the API:

uvicorn app.api:app --reload

Check it's alive:

curl http://127.0.0.1:8000/health

List the available local models:

curl http://127.0.0.1:8000/v1/models
curl http://127.0.0.1:8000/v1/models/llm-lessons

Send a Question

curl -X POST http://127.0.0.1:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "llm-lessons",
    "messages": [
      {"role": "user", "content": "What pizza do you recommend?"}
    ],
    "temperature": 0.1,
    "max_tokens": 120
  }'

What the Request Looks Like

{
  "model": "llm-lessons",
  "messages": [
    {"role": "user", "content": "What pizza do you recommend?"}
  ],
  "temperature": 0.1,
  "max_tokens": 120
}

temperature controls how random the output is. For this tiny model, the API caps it at 0.5 because higher values usually produce noise. max_tokens is also capped at 200 so one request cannot accidentally run thousands of slow character-generation steps. The API also accepts OpenAI-style max_completion_tokens as an alias. Streaming is intentionally rejected because this study model returns one complete local answer.

What the Response Looks Like

{
  "id": "chatcmpl-local",
  "object": "chat.completion",
  "created": 1716300000,
  "model": "llm-lessons",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "I recommend the Margherita pizza."
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 96,
    "completion_tokens": 34,
    "total_tokens": 130
  }
}

This format intentionally mirrors the OpenAI Chat Completions API: model IDs live under /v1/models, chat requests go to /v1/chat/completions, and responses include created, choices, and usage. Usage is approximate here because the study model uses character-level tokens.

Why This Format Matters

Real-world apps are built around the OpenAI API format. By following the same shape, our local model becomes a drop-in replacement — useful for testing, privacy-focused apps, or offline environments.

The API Is Thin By Design

The API doesn't contain any model logic. It:

  1. Receives the request
  2. Validates it
  3. Calls generate_text() from app/infer.py
  4. Wraps the result in the response shape

All intelligence stays in the model pipeline. The API is just a translator between HTTP and inference.

Explore the Auto-Generated Docs

FastAPI generates interactive documentation automatically. While the server is running, open:

http://127.0.0.1:8000/docs

You can test the API directly from your browser — no curl needed.
