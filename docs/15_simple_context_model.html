How a Conversation Becomes Model Input

Before the model can do anything, it needs to read your question. But here's the catch: the model doesn't understand "questions" or "conversations." It only understands a sequence of numbers.

So how does a chat conversation become a number sequence? That's what this lesson is about.

The Model Doesn't See a Chat — It Sees a Blob of Text

When you use ChatGPT, you see a nice chat interface: your message on the right, the AI's reply on the left. Behind the scenes, this project squashes the chat into a single flat string of text, using the same plain labels that appear in the dataset:

Customer: What pizza do you recommend?
Assistant:

The model reads this whole string and continues it. The open Assistant: label at the end is a hint: "now it's your turn to respond."

This formatting is handled by app/context.py.

Why the Format Has to Match the Training Data

Look at app/dataset.txt. Dialogue examples use the same labels:

Customer: What pizza do you recommend?
Assistant: I recommend the Margherita pizza.

The model learned that text after Assistant: is the kind of thing assistants say. So when inference puts an open Assistant: at the end of your prompt, the model naturally continues with assistant-style text.

If training used one format and inference used a different format, the model would be confused. It's like teaching someone to respond to "Marco!" with "Polo!" — then shouting "Hey!" and wondering why they don't respond.

What Is a Token?

Before the text reaches the model, the tokenizer splits it into small pieces called tokens. In this project, each character is one token:

"pizza" → ['p', 'i', 'z', 'z', 'a']

Then each character gets an integer ID from the vocabulary:

['p', 'i', 'z', 'z', 'a'] → [32, 17, 51, 51, 8]

Now the model can process numbers.

In real GPT models, tokens are usually small word-pieces (like "piz" and "za"), not single characters. That's more efficient, but harder to understand. We use characters to make every step visible.

What Is a Context Window?

The model can't look at infinite history. It has a memory limit called the context window.

In this project, block_size = 64 — meaning the model sees at most 64 characters at a time.

This is a long conversation that went on for a while...
                                        ^^^^^^^^^^^^^^
                                        model only sees this part

When a conversation gets too long, the oldest parts get cut off. The model always sees the most recent text.

Truncation, Sliding Windows, and Forgetting

A context window is not memory in the human sense. It is the visible text span the model can attend to right now. If older tokens fall outside that span, the model cannot use them unless they are summarized, repeated, or retrieved again.

tokens = tokenizer.encode(long_conversation)
context = tokens[-block_size:]  # keep the most recent tokens

Why this matters: long conversations can appear to make a model "forget". Usually it did not forget from its weights; the information simply left the current context.

Visual analogy: a conveyor belt moves under a camera. The model only sees what is inside the camera frame. Old tokens slide away to the left.

TechniqueWhat it does
TruncationDrops old tokens when the sequence is too long
Sliding windowKeeps moving the visible window forward
SummarizationCompresses old chat into a shorter note
RAGRetrieves relevant external notes back into context

Real GPT use: commercial models have much larger context windows than this project, but they are still finite. Long-context design, retrieval, and summarization are engineering strategies for keeping the right information visible.

Mini Summary

The model doesn't understand conversations. It understands sequences of numbers. Every message, every role, every line break gets flattened into one long string, then converted to token IDs, then fed into the model.

The structure isn't magic — it's a pattern the model learned from training data.
